# 発表資料：RAG ベース Q&A API システム

> **対象聴衆**: 技術者・チームレビュアー  
> **資料の使い方**: 各セクションの「ポイント」を読み上げながら発表してください

---

## スライド 1｜自己紹介・背景

### タイトル
「AI を活用した社内ナレッジ Q&A システムの構築」

### ポイント

- 本プロジェクトは、**社内ドキュメント（PDF）への質問応答を自動化**することを目的として開発しました
- 従来は PDF を手動で検索し、担当者に問い合わせるコストが発生していました
- AI（OpenAI）と検索技術（FAISS）を組み合わせた **RAG（Retrieval-Augmented Generation）** を採用することで、ドキュメントを根拠とした正確な回答を自動生成できる仕組みを実装しました
- 開発環境は **Python + FastAPI** を中心に、ローカル環境での動作を優先し、最小限の依存で構築しています

---

## スライド 2｜システム概要・目的

### タイトル
「何を作ったか」

### ポイント

- **一言説明**: PDF ドキュメントを知識源とした、根拠付き Q&A API です
- ユーザーが質問を投稿すると、FAISS でドキュメントを検索し、関連チャンクを OpenAI に渡して回答を生成します（RAG 方式）
- Q&A 以外にも、以下の機能を一体で提供します：
  - **アイテム投稿** と **AI による自動要約**
  - **フィードバック収集**（good/bad）と集計
  - **JWT 認証**によるアクセス制御
- フロントエンドは `static/index.html` のみで、バックエンド API との通信はすべて REST で完結します

## スライド 3｜技術スタック・アーキテクチャ

### タイトル
「どう作ったか」

### 技術スタック一覧

| レイヤー | 使用技術 | 役割 |
|--------|---------|------|
| API サーバー | FastAPI 0.136 | REST API・Swagger 自動生成・非同期処理 |
| ORM / DB | SQLAlchemy 2.0 + SQLite | ユーザー・アイテム・フィードバックの永続化 |
| 認証 | python-jose + passlib（bcrypt） | JWT 発行・パスワードハッシュ |
| RAG 検索 | FAISS-cpu + LangChain | PDF ベクトル化・類似検索 |
| AI 生成 | OpenAI API（GPT / text-embedding-3-small） | 回答生成・本文要約・埋め込みベクトル生成 |
| バリデーション | Pydantic v2 | リクエスト/レスポンスの型安全な検証 |
| ロギング | CSV ファイル（`logs/token_usage.csv`） | トークン使用量の記録 |

### アーキテクチャのポイント

- **RAG の仕組み**: PDF を 500 文字のチャンクに分割 → OpenAI Embeddings でベクトル化 → FAISS インデックスに保存。質問時はベクトル類似度で上位 3 チャンクを取得し、OpenAI の回答生成プロンプトに注入します
- **非同期要約**: アイテム投稿後、FastAPI の `BackgroundTasks` を使って要約生成を非同期実行します。ユーザーへのレスポンスは即時返却され、体感速度を損ないません
- **プロンプトの外部化**: `prompts/qa_prompt_v2.yaml` にプロンプトを YAML で管理し、コード変更なしでプロンプトを調整できます
- **最小構成**: DB は SQLite、インフラは uvicorn 単体で動作します。スケール時は PostgreSQL・Docker Compose への移行が容易な設計です

## スライド 4｜デモ説明（正常系・異常系）

### タイトル
「実際に動かすと」

> 詳細な操作手順は [docs/demo_scenario.md](demo_scenario.md) を参照してください

### 正常系のポイント（Swagger: `http://localhost:8000/docs`）

1. **ユーザー登録 → ログイン**: `POST /auth/register` で登録し、`POST /auth/login` で JWT を取得します
2. **アイテム投稿**: JWT を Swagger の「Authorize」に登録し、`POST /items` で投稿します。レスポンスの `summary` は `null` です
3. **AI 要約の確認**: 数秒後に `GET /items` を実行すると、`summary` に AI 生成の要約が入っています
4. **Q&A**: `POST /qa/` に質問を投げると、回答と引用元ドキュメントの情報（ファイル名・ページ・チャンク冒頭）が返ります
5. **フィードバック → 集計**: `POST /feedback/` で good/bad を送り、`GET /feedback/summary` で集計結果を確認します

### 異常系のポイント

| ケース | 操作 | 期待するエラー |
|------|------|-------------|
| 未認証投稿 | JWT なしで `POST /items` | `401 Unauthorized` |
| 不正評価値 | `rating: "excellent"` で `POST /feedback/` | `422 Unprocessable Entity` |
| PDF 外の質問 | 無関係な質問を `POST /qa/` | `200 OK`（AI が「わかりません」と返答） |

- **異常系 A** は認証の必要性を示します。JWT なしでは保護された API へのアクセスを完全に遮断します
- **異常系 B** は Pydantic のバリデーション機能です。不正な値は API 内部に到達する前に弾かれます
- **異常系 C** が最重要です。プロンプトに「コンテキスト外の情報は答えない」ルールを明記しているため、AI がハルシネーション（でたらめな回答）を抑制します

## スライド 5｜運用・コスト管理

### タイトル
「どう運用・管理するか」

### ポイント

- **トークン使用量の記録**: `logs/token_usage.csv` にリクエスト ID・操作名・モデル名・入出力トークン数を自動記録します。コスト増加の原因追跡に使用できます

- **主なコスト発生ポイント**:

| 操作 | モデル | 発生タイミング |
|-----|------|-------------|
| アイテム要約 | GPT（ChatCompletion） | `POST /items` ごと |
| Q&A 回答生成 | GPT（ChatCompletion） | `POST /qa/` ごと |
| FAISS 再構築 | text-embedding-3-small | PDF 追加時のみ |

- **コスト削減の工夫（現在の実装）**:
  - チャンクサイズを 500 文字に制限してトークン数を抑制
  - 類似検索の取得数を `k=3` に制限してプロンプトへの注入量を最小化

- **環境変数・秘密情報の管理**:
  - `OPENAI_API_KEY` と `SECRET_KEY` は `.env` ファイルで管理し、Git 管理外（`.gitignore` に記載）
  - `app.db`（SQLite データファイル）も Git 管理外とし、本番データの誤コミットを防ぎます

## スライド 6｜強みと弱み・今後の改善案

### タイトル
「できていることと、課題」

### 強み

| 強み | 詳細 |
|-----|------|
| 根拠付き回答 | 回答に引用元ファイル・ページが添付されるため、情報の信頼性を確認できる |
| ハルシネーション抑制 | プロンプトでコンテキスト外の回答を禁止しており、PDF にない情報を回答しない |
| 非同期設計 | AI 要約を `BackgroundTasks` で処理するため、API のレスポンス速度に影響しない |
| 品質評価済みプロンプト | `eval/eval_pipeline.py` で v1/v2 を定量比較し、スコアが高い v2 を採用 |
| 最小依存・即起動 | SQLite + uvicorn のみで起動可能。開発・検証が容易 |

### 弱み・課題

| 弱み | 詳細 |
|-----|------|
| SQLite の限界 | 高負荷・同時書き込み環境では競合が発生しうる |
| JWT 秘密鍵がハードコード | `services/auth.py` の `SECRET_KEY` がデフォルト値のため、本番では必ず環境変数化が必要 |
| ユーザーロールなし | 管理者・一般ユーザーの権限分離が未実装 |
| FAISS の手動更新 | PDF 追加時にインデックス再構築スクリプトを手動実行する必要がある |
| テストコードなし | 自動テスト（pytest）が未整備 |

### 今後の改善案のポイント

- **短期**:
  - `SECRET_KEY` を環境変数化して本番セキュリティを確保
  - `pytest` によるユニットテスト・統合テストを追加
  - JWT 有効期限のカスタマイズを環境変数で制御
- **中期**:
  - ページネーション（`GET /items?offset=0&limit=20`）を追加
  - FAISS 再構築を管理者 API エンドポイント化
  - `gpt-4o-mini` 等の軽量モデルを要約に使用してコスト削減
- **長期**:
  - PostgreSQL への移行とスケールアウト対応
  - Docker Compose で API・DB・Nginx を一括コンテナ化
  - GitHub Actions による CI/CD パイプライン構築

## 想定質疑応答

### 技術に関する質問

---

**Q1. RAG と通常のプロンプトだけの Q&A の違いは何ですか？**

> **A.**
> 通常のプロンプトのみでは、OpenAI の学習データに含まれない社内固有のドキュメント情報は回答できません。RAG では質問に関連するチャンクを FAISS で検索してプロンプトに動的に注入するため、**社内 PDF の最新内容を根拠にした回答**が可能です。また、引用元を回答に添付することで、回答の根拠を人間が検証できます。

---

**Q2. FAISS インデックスはどのように作成・更新しますか？**

> **A.**
> `rag/rag_pipeline.py` を実行することで、`data/` フォルダ内の PDF を読み込み、500 文字チャンクに分割→ OpenAI Embeddings でベクトル化→ `rag/faiss_index/` に保存します。PDF を追加した場合はスクリプトを再実行してインデックスを再構築します。現在は手動実行ですが、管理者 API 化や CI トリガーへの拡張を検討しています。

---

**Q3. プロンプトのバージョン管理はどうしていますか？**

> **A.**
> `prompts/` フォルダに `qa_prompt_v1.yaml`・`qa_prompt_v2.yaml` として YAML ファイルで管理しています。`eval/eval_pipeline.py` を実行すると同一の 5 問でバッチ評価を行い、正確性・簡潔性・根拠・総合の 4 指標でスコアを算出して CSV に保存します。最新評価では v2 の総合スコアが `3.65`（v1 は `3.47`）と高く、特に簡潔性が `2.9 → 3.5` に改善したため v2 を採用しています。

---

### 運用に関する質問

---

**Q4. 本番環境に展開する場合、何を変更する必要がありますか？**

> **A.**
> 最低限必要な変更は以下の 3 点です。
> 1. **`SECRET_KEY` の環境変数化**: 現在 `services/auth.py` にデフォルト値が書かれているため、環境変数 `SECRET_KEY` から読み込む形に変更します
> 2. **SQLite → PostgreSQL の移行**: 接続 URL を `DATABASE_URL` 環境変数で切り替えるだけで対応可能な設計です
> 3. **CORS の制限**: 現在フロントエンドのオリジンを制限していない場合、本番ドメインのみを許可するよう設定します
>
> 加えて HTTPS 化（nginx + Let's Encrypt）と Docker Compose によるコンテナ化を推奨します。

---

**Q5. ユーザーのパスワードはどのように保護されていますか？**

> **A.**
> `passlib` の bcrypt アルゴリズムでハッシュ化して SQLite に保存しています。平文パスワードは DB に一切保存されません。bcrypt はソルト付きのハッシュを生成するため、同じパスワードでも毎回異なるハッシュ値になり、レインボーテーブル攻撃に強い特性があります。JWT の有効期限は 30 分に設定されており、トークン漏洩時の被害を最小化します。

---

**Q6. ログはどこで確認できますか？**

> **A.**
> `logs/token_usage.csv` にリクエスト ID・操作名・使用モデル・入出力トークン数がタイムスタンプ付きで記録されます。Excel や pandas で集計することでエンドポイントごとのコストを可視化できます。アプリケーションエラーログは uvicorn の標準出力に出力されるため、本番環境では systemd や Docker のログドライバーで収集することを推奨します。

---

### コストに関する質問

---

**Q7. 月額コストの目安はどのくらいですか？**

> **A.**
> 使用量に依存しますが、主要なコスト要因を示します。
>
> | 操作 | モデル | 概算コスト |
> |-----|------|---------|
> | アイテム要約（1 回） | GPT-4o | 約 $0.001〜$0.003 |
> | Q&A 回答（1 回） | GPT-4o | 約 $0.002〜$0.005 |
> | FAISS 再構築（100 ページ PDF） | text-embedding-3-small | 約 $0.001 |
>
> 社内利用で 1 日 100 件の Q&A を想定すると月額 $10〜20 程度の見込みです。要約に `gpt-4o-mini` を使用することで、要約コストを約 1/15 に削減できます。

---

**Q8. コストが急増した場合の検知方法はありますか？**

> **A.**
> 現状は `logs/token_usage.csv` を手動集計する形です。改善案として、以下を検討しています。
> 1. **日次レポートスクリプト** (`scripts/daily_report.py`) による CSV の自動集計と Slack 通知
> 2. **OpenAI ダッシュボードのアラート設定**（月次の上限金額を設定し、超過時にメール通知）
> 3. **トークン数の上限バリデーション**（リクエスト本文の最大文字数を API 側で制限）

---

**Q9. OpenAI 以外の AI サービスに切り替えることはできますか？**

> **A.**
> LangChain を介して OpenAI を呼び出しているため、LangChain が対応している他のプロバイダー（Azure OpenAI・Anthropic Claude・Google Gemini 等）への切り替えは比較的容易です。`ChatOpenAI` クラスを対応する LangChain クラスに差し替え、埋め込みモデルも対応クラスに変更することで対応できます。ただし、プロンプトの最適化は各モデルの特性に合わせて再評価が必要です。
