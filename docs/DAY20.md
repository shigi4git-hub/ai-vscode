・仮想環境（venv）を使う理由 深掘り
　プロジェクトごとに「専用の部屋」を作り、
　ライブラリの衝突や汚染を防ぐため。

【具体的な問題：venvなしの場合】

　venvを使わないと、すべてのライブラリが
　PC全体（グローバル環境）に入る。

　例として、プロジェクトAとBで、fastapiのバージョン違いを設定できる。
  プロジェクトA → fastapi 0.80 が必要
  プロジェクトB → fastapi 0.95 が必要
　※同じPCに両方は入れられないため、どちらかが動かなくなる。

【venvを使うと解決する】

  プロジェクトA/.venv/ → fastapi 0.80 だけ入っている
  プロジェクトB/.venv/ → fastapi 0.95 だけ入っている

　それぞれ完全に独立しているため、互いに影響しない。

【.gitignoreに .venv/ を書く理由】

　.venv/ フォルダは容量が大きい（数百MB）。
　また、環境ごとに再作成できるため、
　GitHubに上げる必要がない。

　他の人がリポジトリをcloneしたときは、
　pip install -r requirements.txt を実行すれば
　同じ環境を再現できる。

　→ つまり「ライブラリの実体」ではなく
   「ライブラリのリスト（requirements.txt）」だけ
  　 GitHubで管理するのが正しい作法。

【まとめ】

　venvを使う理由は3つ：

　1. プロジェクト間のライブラリバージョン衝突を防ぐ
　2. グローバル環境を汚さない
　3. requirements.txt で環境を再現できるようにする

【管理者質問への回答】
Q. JSONの構造を説明できるか？
A. JSONとは、システム間でデータをやり取りする際の共通フォーマット。
キーと値のペアで構成され、文字列・数値・真偽値・配列・オブジェクト・nullを扱える。
オブジェクトの中にオブジェクトを入れる入れ子構造（ネスト）も可能で、
複雑なデータ構造を表現できる点がCSVとの大きな違い。
が、WEBとAIで出た回答でした。

JSONファイルは知っておりましたが、CSVで今まで対応してました。

CSV（表形式）の場合、１行目に項目を設定があり、それに沿ってデータ作成をする。
苗字, 名前, 住所
田中, 太郎, 東京
山田, 花子, 東京

JSONの場合は、構造でデータ作成ができる。
また、キーの並び順は関係ない。
{"名前": "太郎", "苗字": "田中", "住所": "東京"}
{"苗字": "田中", "住所": "東京", "名前": "太郎"}
→ 上記2つは同じデータとして扱われる。入れ子構造（ネスト）

更に、CSVファイルではできない階層構造も可能。
{ "苗字": "田中",  "住所":  { "都道府県": "東京", "区": "台東区", "名前": "太郎" }
と捉えました。
報告書のため、１行で書いているが、項目毎に改行とインデントの複数行で管理。

今回のPOST /items では、titleとbodyをJSON形式で受け取り、
201レスポンスとしてJSONを返す実装を行った。

Day 2で、返す（GET）。本日Day 3は、受け取る（POST）。体現した。
Day 4（SQLite/ORM）に向けて、データの永続化処理の実装を進めてまいります。

案件参画を想定し、基礎知識の整理と理解を進めてまいります。

□ 本日のつまずきポイント・備忘録
つまずきポイント：
　Copilotが生成したコードでは空文字が通ってしまった。
　str型は空文字を許可するため、Field(min_length=1)を追加して対応した。
　また、Fieldのimportが漏れており、NameErrorが発生した。

【用語メモ】
1.Pydanticとは
　 Pythonのデータバリデーションライブラリ。
　 定義した型と異なるデータが来た場合、自動で422エラーを返してくれる。
　 FastAPIと組み合わせることで、不正なデータをAPI層で弾くことができる。
2.バリデーションとは
　 入力データが正しいかチェックすること。
　 例）title: str → 文字列じゃないとダメ。
　 ただしstr型のみでは空文字は弾けないため、Field(min_length=1)を追加する必要がある。
3.JSONとは
　 システム間でデータをやり取りする際の共通フォーマット。
　 CSVは表形式のため入れ子構造が作れないが、
　 JSONはオブジェクトの中にオブジェクトを入れる入れ子構造（ネスト）が可能で、
　 複雑なデータ構造を表現できる。
4.正常系・異常系とは
　 正常系：想定通りの入力でAPIが正しく動くことを確認するテスト。
　 異常系：不正な入力でAPIが正しくエラーを返すことを確認するテスト。
5.201レスポンスとは
　 HTTPステータスコードの一つ。
　 「データの作成が成功した」を意味する。
　 200（成功）とは異なり、新しいリソースが作成された場合に使う。
6.Fieldとは
　 Pydanticのフィールドに追加のルールを設定するための機能。
　 例）Field(min_length=1) → 1文字以上でないとNG。
　 使用するにはfrom pydantic import BaseModel, Field のimportが必要。

【管理者質問への回答】
Q. ORMを使う利点は？
A. ORMを使うと、SQLを書かずにPythonだけでDBを操作できる。
SQLAlchemyが裏側でSQLに変換してくれるため、Pythonのコードだけでデータの保存・取得・削除ができる。
またDBの種類（SQLiteやPostgreSQL）が変わってもPythonのコードはほぼそのまま使えるため、変更に強い。

深堀を実施。

ORMは、PythonのクラスとDBのテーブルを対応させて、SQLを書かずにPythonだけでDBを操作できるようにするツール。
今回SQL文を記載していないにもかかわらず、DBにデータが保存されていた。
今回の場合は、database.pyにコードは記載しており、DBに対応しているが、SQL文ではない。
長いSQL文だと、間違い探しに時間がかかるが、構造的なコードで記載されている点が、視覚的に良いと感じた。
また、コードを記載しておけば、他のDBに変更しても修正が軽微で済む点が良いとも感じた。
一言で言うとPythonとDBの「通訳」と言うことで、捉えました。

主なORMの種類として、
SQLAlchemy：Python　本日の学習で使用。高機能で本番でも多く使われるとのこと。FastAPIとの組み合わせが一般的。
Prisma：JavaScript/TypeScript　Node.js系で使用頻度が高い。
Hibernate：Java　Java系の定番である。
また、
Django ORM：Python　DjangoというWebフレームワークに内蔵であるため、
本日の学習では、Pythonではあるが、フレームワークがFastAPIであったため不使用。

【管理者質問への回答】
Q. ハッシュ化と暗号化の違いは？
A. ハッシュ化は元に戻せない一方向変換。暗号化は鍵があれば元に戻せる双方向変換。
パスワード保存にハッシュ化を使う理由は、DBが漏洩しても元のパスワードが復元できないため。
なお、ハッシュ値は元のパスワードに戻すためではなく、ログイン時の「照合」に使用する。
ログイン時は入力されたパスワードを同じハッシュ関数で変換し、DBのハッシュ値と一致するか比較する。

暗号化とハッシュを同等と勘違いするため、まとめ直すと、
暗号化の鍵 ＝ 元のデータに「戻すため」のもの
ハッシュ値 ＝ 元のデータに「照合するため」のもの
更に、機能的には似ています。ただ厳密には、
鍵 ＝ 元に戻せる
ハッシュ値 ＝ 元に戻せない
この「戻せるか戻せないか」が唯一の違いがあると捉えました。


【本日のつまずきポイント・備忘録】

つまずきポイント：

【用語メモ】

1.ハッシュ化とは
元に戻せない一方向変換。パスワード保存に使用する。
bcryptはハッシュ化アルゴリズムの一つ。DBに$2b$12$...という形式で保存される。

2.passlibとは
Pythonのパスワードハッシュ化ライブラリ。bcryptと組み合わせて使用する。
CryptContextでハッシュ化のアルゴリズムを設定する。

3.暗号化とは
鍵があれば元に戻せる双方向変換。データの送受信に使用する。
ハッシュ化との違いは「元に戻せるかどうか」。

4.HTTPException とは

【管理者質問への回答】
Q. JWTの構成要素を説明できるか？
A. JWTはヘッダー・ペイロード・署名の3つで構成されており、ドット（.）で区切られている。

JWTについては、今回初見だったため、以下の通り、内容を確認しました。
ヘッダーには、署名に使うアルゴリズム情報（今回はHS256としました）が含まれる。
ペイロードには、ユーザーIDや有効期限が含まれる。
署名には、改ざん防止のための情報が含まれる。
今回Swaggerで発行されたトークンを確認すると、eyJhbGci...（ヘッダー）.eyJzdWIi...（ペイロード）.1SgOfs...（署名）という形式になっており、3つの構成要素を実際に目で確認した。
なお、ペイロードはBase64形式でエンコードされているだけのため、jwt.ioで中身を確認できる。秘密情報をペイロードに入れてはいけないと理解した。
まとめると、JWTはドット（.）で3つに分かれている。
　① ヘッダー　　：アルゴリズム情報（HS256など）
　② ペイロード　：ユーザーID・有効期限
　③ 署名　　　　：改ざん防止

健康診断（通行手形）で例えると、
① 窓口で予約表と保険証（本人確認で免許）を見せる　→　ログイン
② スタッフが患者用リストバンドをくれる　　→　JWT発行
③ 以降は患者用リストバンドを見せるだけで検査を受診　→　JWT提示
④ 最後の会計等に保険証等はもう一度見せなくていい →　毎回ログイン不要
※ 有効期限は、当日まで有効。
上記のように、捉えました。

JWTはログイン機能としてだけでなく、将来の自律型AIエージェントがユーザーを識別するための基盤にもなると理解した。

タイムアウト・例外処理も実装し、エラー時はエラーイベントを送信する形で対応しています。

【管理者質問への回答】
Q. 同期と非同期の違いは？
A. 同期とは、処理が完了するまで次の処理に進めない方式。処理が順番に一つずつ実行されるため、待ち時間が発生する。非同期とは、処理の完了を待たずに次の処理を進められる方式。待ち時間の間に別の処理を実行できるため、効率が良い。今回のストリーミングに当てはめると、同期は「AIが全文を生成し終わるまで待ってから一気に表示」、非同期は「AIが生成しながら少しずつ送ってきて、届いた順にリアルタイムで表示」となる。UXの観点では、同期の場合はユーザーが数秒間「何も起きていない」と感じるため、離脱やストレスにつながりやすい。非同期（ストリーミング）の場合は、回答が流れてくることで「AIが動いている」と視覚的に伝わり、待ち時間のストレスが大幅に軽減される。生成AIがストリーミングを採用しているのも、このUX向上が理由と理解した。コードではasync defが非同期関数の宣言、await asyncio.sleep(0.01)がチャンク送信後に他の処理へ切り替えるタイミングを示している。
まとめると、今まで（Day5・6）のfetchは「全部届いたら一気に受け取る」でした。今日のstream readerは「届くたびに少しずつ受け取る」に変更している。が、AIとの壁打ちで分かった回答となります。
個人的なイメージでは、通販サイトで複数点の商品を注文した際、同期（fetch）は全商品が揃ってから一度にまとめて発送・受け取ること。非同期（stream reader）は準備できた商品から順番に発送され、届くたびに受け取ることと捉えました。


【本日のつまずきポイント・備忘録】
つまずきポイント：特になし。
※但し、新しい用語等に少し手こずりました。イメージも掴み辛かったです。


【用語メモ】
1.ストリーミングとは
データを少しずつ送り続ける方式。生成AIで回答が少しずつ表示されるあの動きのこと。全部生成し終わるまで待つのではなく、生成しながら届いた順に表示できる。

2.SSE（Server-Sent Events）とは
サーバーからクライアントへ一方的にデータを送り続けるHTTPの仕組み。今回のストリーミングで使用した形式。media_typeはtext/event-streamを使用する。

3.StreamingResponseとは
FastAPIの機能。データを少しずつ返すためのレスポンス形式。通常のJSONResponseは一括で返すのに対し、StreamingResponseは逐次返却できる。

4.async / awaitとは
Pythonの非同期処理の書き方。async defで非同期関数を宣言し、awaitで他の処理に切り替えるタイミングを指定する。今回はawait asyncio.sleep(0.01)でチャンク送信後に短い待機を入れている。

5.チャンクとは
データを分割した小さな塊のこと。ストリーミングでは、AIの回答をチャンク単位で受け取り、届いた順に画面に表示する。
トークンはAI内部の単位、チャンクは通信上の単位です。

【管理者質問への回答】
Q. 埋め込み（Embedding）とは何か？
A. 文章や単語を数値のリスト（ベクトル）に変換すること。例えば「犬」と「猫」は意味が近いため数値も近くなり、「車」は意味が遠いため数値も遠くなる。今回のRAGに当てはめると、PDFの45チャンクをそれぞれ数値のリストに変換してFAISSに保存したのがSTEP3。検索時は質問文も同じように数値に変換し、数値が近いチャンクを探すことで類似検索を実現している。一言でまとめると「意味の近さを数値で表す仕組み」と理解した。が、AIとの壁打ちで分かった回答となります。
AIとの壁打ちで得たイメージは、色のRGB値に近いと捉えました。赤（255,0,0）と橙（255,165,0）は数値が近く、青（0,0,255）は数値が遠い。同じように、意味が近い言葉は数値が近くなり、意味が遠い言葉は数値が遠くなる。埋め込みはこのRGBのように、言葉の意味を数値で表現したものとAIと壁打ちで得た回答になります。

個人的な感覚では、「夜空、星、月」は、近い言葉（ベクトル）であること。しかし、文章にした場合の類似検索において、
A：夜空が綺麗だ。
B：星が良く見える。
C：月見バーガーの季節だ。
AとBは、比較的近いベクトルとなる可能性があり、Cは文章の意味としては遠いベクトルになる可能性があると捉えました。

## Day 15 - RAG(検索)

GitHub CopilotにてRAGの回答生成パイプラインをrag/rag_pipeline.pyに追加し、app/routers/qa.pyを新規作成しました。
POST /qaエンドポイントに質問を投げると、昨日作成したFAISSインデックスから関連チャンクを検索し、GPT-4o-miniで回答を生成して引用元chunk（source・page・内容の冒頭）とともに返却する実装です。
「AIとは何ですか？」という質問に対して、test.pdf（JSTのAI活用ガイド）の内容に基づいた正確な回答と引用元が返されることを確認しました。
また、PDFに存在しない「本日（2026/6/17）の埼玉県の天気は？」という質問に対して「わかりません。」と回答され、根拠にない情報は回答しないプロンプト設計が正しく動作していることを確認しました。

【管理者質問への回答】
Q. RAGで最も重要な工程は？
A. 検索（Retrieval）が最も重要。検索で間違ったチャンクを取ってきた場合、どんなに良いプロンプトを書いても回答が間違うため。今日の実装で言うと、「AIとは何ですか？」の質問に対して正しいチャンク（page1のAIの定義部分）が取れたから正しい回答が生成できた。一言でまとめると「検索精度が土台、プロンプトが仕上げ」と理解した。また、プロンプト設計も重要で、「根拠にない情報は回答しない」という指示がなければ、PDFにない天気の質問でハルシネーションが起きていた可能性がある。が、AIとの壁打ちで分かった回答となります。
上記の回答をもとに、咀嚼した内容は、
バグだらけのコード修正において、プログラミング力がある開発者において、違う仕様書を渡すと間違った修正を実施してしまう。また、ただ、バグ修正しておいてと言われても、欠陥を修正するが、必要な部分も削除してしまう可能性がある。
・違う仕様書を渡す → 間違ったPDFを読み込ませる（ゴミが入る）
・間違った修正をする → 間違った回答が返ってくる（ゴミが出る）
・「バグ修正しておいて」だけ → プロンプトが曖昧（ルールがない）
・（曖昧な指示のため）必要な部分も削除してしまう → ハルシネーション（根拠なく回答する）
RAGの「ゴミが入ればゴミが出る」の設計思想で、RAGも開発も「正しい情報と明確な指示がなければ正しい結果が出ない」という同じ構造と捉えました。

【本日のつまずきポイント・備忘録】
つまずきポイント：特になし。

【用語メモ】
1.RAGの流れとは
質問をベクトル化 → FAISSで類似チャンク検索（Retrieval） → コンテキスト組立（Augmented） → LLMで回答生成（Generation）という一連の流れ。

2.Retrieval（検索・取得）とは
RAGの「R」にあたる工程。質問をベクトル化し、FAISSインデックスから関連チャンクを取得する処理。検索精度がRAG全体の品質を左右するため最も重要な工程。

3.コンテキストとは
LLMに回答生成の根拠として渡す情報のこと。今回は検索で取得したチャンクをコンテキストとしてプロンプトに組み込んでいる。

4.ハルシネーションとは
AIが根拠のない情報を事実かのように回答してしまう現象。RAGでは「根拠にない情報は回答しない」とプロンプトで明示することで抑制できる。

5.引用元chunk（sources）とは
RAGの回答に付与する根拠情報。どのファイルの何ページのどの内容を根拠に回答したかを示す。回答の信頼性を高めるために重要。

【ループエンジニアリング】
Day19‐フィードバックの簡易的なフィードバックループを自動化する簡易実装を行いました。
タイマー式ではなく、Bad評価時、フィードバックループ（3回まで）を実行するようにしております。

実装内容は以下の通りです。

・static/feedback_loop.html を新規作成し、チャット形式のUIを実装しました。質問を入力するとPOST /qaで回答を取得し、各回答の下にGOOD・BADボタンを表示します。

・prompts/feedback_loop/フォルダを新規作成し、normal_prompt.yaml（初回質問用・qa_prompt_v2.yamlと同内容）とretry_prompt.yaml（BAD時の再質問用・「前回の回答では不十分でした。より詳しく説明してください。」を追加）の2つのプロンプトを分離管理しました。既存のqa_prompt_v2.yamlは変更していないため、既存環境に影響しない設計としました。

・app/services/token_logger.py にlog_feedback_loop関数を追加し、logs/feedback_loop_log.csvにtimestamp・request_id・question・prompt_type・使用したプロンプト内容・answer・rating・loop_count・input_tokens・output_tokensを記録する仕組みを実装しました。

・app/routers/qa.py を修正し、POST /qaにprompt_type（normal/retry）とloop_count（int）を追加しました。prompt_type未指定時は既存のqa_prompt_v2.yamlを使用する設計で、既存の動作を保持しています。
動作確認として、「AIとは何ですか？」と質問し、以下を確認しました。

・1回目（normal）：簡潔な回答が返された

・BAD押下後（retry・loop_count:1）：より詳しい回答が返された

・BADを3回連続で押すと「改善できませんでした。別の質問をお試しください。」と表示されループが終了した

・GOOD押下時は「回答に満足いただきありがとうございます！」と表示されループが終了した

・元ネタが少ないため、改善されたかどうかの判断が難しいかったです。より詳しくなっているが、長文になったり、ほぼ内容が同じで、ピックアップする部分と構成が変わっただけのような気がします。

・logs/feedback_loop_log.csvにprompt_typeとloop_countが正常に記録されていることを確認した
今回の実装で、「やってみる→評価する→改善して再試行する」というフィードバックループの基本構造をシステムとして体現しました。社長からご共有いただいたループエンジニアリングの「人間が設計したフローをAIに自動でやらせる」という考え方を、簡易版として実装できたと考えています。
